# :cooking: System Design: Twitter

A __Twitter__ system design focuses on building a highly scalable, low-latency social feed platform where users can post short messages (tweets), follow others, and view personalized timelines.

<!-- more -->

At a high level, the system includes core services for user management, tweet creation, follow relationships, and timeline generation. Tweets are stored in a distributed database optimized for fast writes, while user and follow data support graph-like relationships.

The key design challenge is the feed (timeline) generation, which is typically handled in two ways:

- Fan-out on write: When a user posts a tweet, it is pushed to all followers’ timelines in advance. This makes reading fast but can be expensive for users with many followers.
- Fan-out on read: Tweets are fetched and merged at request time when a user opens their timeline. This reduces write cost but increases read latency.
- Most real systems use a hybrid approach depending on user scale (normal users vs celebrities).

To support scalability, the system uses:

- Caching layers (e.g., Redis) for timelines and hot tweets
- Message queues (e.g., Kafka-like systems) for async processing of tweets and fan-out
- Sharded databases to distribute user and tweet storage
- CDN for media delivery (images/videos)

Search functionality is handled via a separate indexing system (like Elasticsearch), and trending topics are computed via real-time analytics pipelines.

Finally, the system is designed to handle high read-to-write ratios, eventual consistency in feeds, and horizontal scaling across all major components.
